## 二、详细问题描述

### P0-1 项目内工具 skill 检索不全

- **模块**：工具检索逻辑
- **现象**：以 Wiki Project 为例，Cloude 工具下实际共有 **12 个 skill**，但在 Studio 的扫描界面中只显示了 **10 个**，漏掉了 2 个。
- **复现步骤**：
  1. 打开 Wiki Project；
  2. 进入 Studio 扫描界面，查看 Cloude 工具的 skill 列表；
  3. 对照实际目录，发现数量不一致（12 vs 10）。
- **期望行为**：扫描结果应完整枚举项目内该工具下的全部 skill，数量与磁盘实际一致。
- **实际行为**：稳定少显示 2 个，疑似检索有上限、分页丢失、或过滤/去重逻辑误删。
- **建议排查方向**：
  - 检索是否存在硬上限（如默认取前 N 条）；
  - 是否存在"已备份到 SSOT"就被从项目列表过滤的逻辑；
  - 同名/软链接 skill 是否被去重误删。

---

### P0-2 项目级更新路径错误（显示全局路径）

- **模块**：更新路径解析
- **现象**：在 Wiki Project 中执行"检索更新"时，报出的更新路径指向 **全局 WorkBuddy 路径（C 盘下的全局目录）**，而非该项目自身的路径。
- **复现步骤**：
  1. 进入 Wiki Project；
  2. 触发"检索更新 / 检测差异"；
  3. 查看弹出的路径提示，发现是 `C:\Users\...\WorkBuddy\...` 全局路径。
- **期望行为**：项目级操作应解析并显示**项目本地路径**（即 Wiki Project 所在目录下的 skill 路径）。
- **实际行为**：错误地使用了全局路径，导致用户无法确认到底更新的是哪一份文件，存在改错文件的风险。
- **建议**：项目级扫描/更新时，路径解析应以"当前项目根目录"为基准，而非回退到全局目录。

---

### P0-3 检测机制按表单顺序逐个检测，导致循环误报

- **模块**：差异检测调度逻辑
- **现象**：检测不是"全量扫描所有工具 → 统一找差异"，而是**按表单顺序逐个检测**；更新其中一个工具到 SSOT 后，再次检测仍绕回前面工具，形成循环提示。
- **复现步骤（用户实测）**：
  1. 在 Claude Code 和 Codex 的 Classify skill 中各有改动；
  2. 首次检测：按顺序先查 Claude Code，报"Claude Code 有差异"，此时**未报告出 Codex 的变化**；
  3. 将 Claude Code 同步到 SSOT，再次检测：仍按 CLINE → Gemini → Codex 顺序；**Codex 的差异未被提供**；
  4. 检测又报"Gemini 有可用更新"；跳过再次检测仍提示 Gemini；
  5. 将 Gemini 同步到 SSOT，再次检测：又变回"Claude Code 的工具有问题"。
- **期望行为**：
  - 先**全量检索所有工具**，**全部扫描完成后**再统一计算各工具与当前 SSOT 的差异；
  - 一次性列出所有有差异的工具，让用户统一决策，而不是边查边报、逐个打断。
- **实际行为**：边检测边提示，且每次只暴露当前表单顺序下的第一个差异，后续工具的差异被掩盖，造成"按下葫芦浮起瓢"的循环。
- **建议**：将"采集阶段"与"决策阶段"分离——先批量采集所有工具当前内容，再批量比对，最后统一展示差异清单。

---

### P1-1 全局与项目级同名 skill 的 SSOT 备份冲突

- **模块**：跨域备份 / SSOT 中心
- **现象**：全局目录下的 skill 备份到 SSOT（单一真理源）；若项目内存在**同名 skill**，二者是否共享同一个 SSOT 中心备份路径，存在错位/互相覆盖的风险。
- **场景**：
  - 全局：`~/.workbuddy/skills/Foo`
  - 项目：`{project}/.workbuddy/skills/Foo`（同名）
  - 两者备份时是否落到同一 SSOT 条目？
- **期望行为**：需明确"同名 skill 的域隔离策略"——全局与项目级应作为**不同域**分别管理，SSOT 路径应带域前缀（如 `global/Foo`、`project:WikiProject/Foo`），避免互相覆盖。
- **待确认**：SSOT 是否以"名称"为唯一键？若是，则同名必然冲突，需改为"域+名称"复合键。
- **建议**：在文档/设计中明确冲突处理规则：同名时以项目级覆盖全局？还是拒绝备份并提示？至少要让用户**可见**当前落在哪个 SSOT 路径。

---

### P1-2 差异对比不直观（整篇被判为新增）

- **模块**：差异计算算法
- **现象**：在 front matter 下方**插入两行**后，系统把**整篇内容都标记为新增**——上方显示 SSOT 原版，下方显示"加了 2 行的新版"，信息量过大。
- **复现步骤**：
  1. 取 SSOT 中的 Classify skill；
  2. 在其 front matter 下方插入 2 行；
  3. 触发差异对比。
- **期望行为**：应像 VSCode 一样，只高亮"此处新增 2 行"，其余内容保持稳定匹配，不被整体重判。
- **实际行为**：因顶部位移导致后续所有行行号错位，整篇被标为新内容。
- **建议**：使用**基于内容的 diff 算法**（如 Myers / LCS / 语义分块），而非纯行号对齐；对 front matter 与正文分块比较，减少位移放大效应。

---

### P2-1 差异查看窗口过小

- **模块**：差异展示 UI
- **现象**：Studio 中"查看具体差异"的弹窗/面板尺寸偏小，长文件难以阅读。
- **期望行为**：窗口应可放大（如支持拖拽边缘、全屏按钮、或默认占更大比例）。
- **建议**：提供最大化/全屏按钮，并记忆用户上次窗口尺寸。

---

### P2-2 缺少 Git / VSCode 式左右分栏对比

- **模块**：差异展示布局
- **现象**：当前差异展示未采用"左=原版（SSOT）/ 右=当前版本"的经典双栏布局。
- **期望行为**：
  - 左侧：SSOT 原版；
  - 右侧：当前工具中的版本；
  - 两栏同步滚动，变更行高亮，新增/删除一目了然。
- **建议**：采用标准双栏 diff 视图（类似 VSCode / GitHub diff）。

---

### P2-3 "查看差异"应支持按工具展开 + 左视图同步定位

- **模块**：差异交互
- **现象**：缺少一个"总览所有差异工具 + 逐工具展开定位"的清晰入口。
- **期望行为**（用户期望的交互模型）：
  1. 提供统一的"**查看差异**"入口，清晰列出**每一个有差异的工具**；
  2. 每个工具是一个**可展开块**：
     - 展开后能看到该工具具体哪一处发生了变化；
  3. **左视图随选中工具同步变化**：
     - 例如 Claude Code 排在最前 → 展开时左视图（SSOT 原版）也定位到最上面的对应块；
     - Codex 排在最后 → 点开 Codex 视图时，左视图同步滚动/定位到 Codex 对应的位置。
- **目标**：用户在一个界面内即可纵览"哪些工具改了、各自改了哪"，无需反复检测、反复跳转。

---

## 三、建议的修复优先级路线

1. **先修数据正确性类（P0）**：P0-1 检索不全、P0-2 路径错误、P0-3 检测循环——这三类直接影响"用户到底改了哪份文件"，属于可信度底线。
2. **再修隔离与算法（P1）**：P1-1 同名冲突（防止数据丢失）、P1-2 差异算法（提升可用性）。
3. **最后做体验增强（P2）**：窗口放大、左右分栏、可展开定位——在 P0/P1 正确的基础上优化交互。
